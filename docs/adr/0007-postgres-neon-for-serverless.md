# ADR 0007: Postgres (Neon) via node-postgres for serverless deployment

**Status:** Accepted (2026-07)

Supersedes the storage-driver decision in [ADR 0002](0002-sqlite-plus-drizzle.md) and
relaxes the runtime constraint behind [ADR 0001](0001-bun-package-manager-node-runtime.md).
The schema-application detail in the Decision section below (`drizzle-kit push`) was itself
later superseded by [ADR 0008](0008-versioned-migrations-on-deploy.md).

## Context

We want to deploy Mercury on Vercel. Vercel's serverless/Fluid Compute filesystem is
read-only except `/tmp`, which is ephemeral and per-instance — a local `better-sqlite3`
WAL file (`data/mercury.db`) cannot persist there, and better-auth stored its
`user`/`session`/`account` rows in that same file. ADR 0002 anticipated this: "Serverless
hosting (Vercel) is off the table without swapping the driver (Turso/libsql or Postgres) —
Drizzle keeps that migration mechanical."

## Decision

Migrate storage to **Neon Postgres**, accessed through **`node-postgres` (`pg`) +
`drizzle-orm/node-postgres`**.

- **Why Neon over Turso/libsql:** it is Vercel's tightest-integrated database — per-preview
  database branching, true concurrency, standard Postgres as a durable foundation. The cost
  (a dialect rewrite of the schema) is a bounded, one-time change.
- **Why `node-postgres` over `neon-serverless`:** one driver works across local Docker/Homebrew
  Postgres, the CI service container, and Neon prod; it supports the interactive transaction
  the SM-2 grade path needs; and on Fluid Compute (which keeps instances warm) it pairs with
  Neon's **pooled** (`-pooler`, PgBouncer) connection string. `pg` is pure JS, so — unlike
  better-sqlite3 — it loads under Bun too.

The migration surface was small: `sqliteTable`→`pgTable` with `timestamp_ms`→`timestamptz`,
`json`→`jsonb`, `real`→`doublePrecision`; the auth schema regenerated by `@better-auth/cli`
for `provider: "pg"`; and one synchronous `db.transaction` (`gradeCard`) rewritten to the
async form (pg-core has no `.get()/.run()` terminals). All other queries were
dialect-agnostic; the `random()` order-by and every `onConflict*` target are portable
as-is. Schema still applies via `drizzle-kit push`; content still seeds by stable slug id.

## Consequences

- Persistence survives Vercel's ephemeral filesystem; sessions and progress are durable.
- `DATABASE_URL` replaces `MERCURY_DB_PATH`. Production uses Neon's pooled string (which
  carries `sslmode=require`); dev/CI/e2e use a local/containerized Postgres.
- **Local dev and CI/e2e now need a real Postgres** (docker-compose, a system Postgres, or a
  Neon branch) — a SQLite file could be a throwaway; Postgres cannot. E2e resets via
  `DROP SCHEMA public CASCADE` (`scripts/db-reset.ts`) instead of deleting a file.
- The **better-sqlite3 napi constraint (ADR 0001) is gone**: `pg` loads under Bun, so
  DB-touching code no longer _crashes_ under the Bun runtime. We keep Bun-install / Node-Next
  and DB-free unit tests as conventions (hermetic, fast tests; parity with Vercel's Node build),
  not hard requirements.
- Postgres removes the single-writer limitation; the exam actions' interleave guards remain
  correct and are now backed by real row-level concurrency.
