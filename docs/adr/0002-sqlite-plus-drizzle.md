# ADR 0002: SQLite + Drizzle ORM

**Status:** Superseded by [ADR 0007](0007-postgres-neon-for-serverless.md) (2026-07) — the
storage driver moved to Neon Postgres (`node-postgres`) for Vercel deployment. The Drizzle
choice and the schema/seed model below still stand; only the SQLite/better-sqlite3 driver
was swapped, as this ADR anticipated.

**Original status:** Accepted (2026-07)

## Context

The first version targets a single-node deployment (VPS/Docker) with no ops budget. Progress data is relational and per-user; content is seeded and read-heavy.

## Decision

SQLite (better-sqlite3, WAL mode) with Drizzle ORM. The schema keeps structured payloads (questions, scripts, exam sections, AI feedback) in JSON columns typed via `$type<>()` against zod-inferred types, so content files, seed validation, and queries share one type source. Schema changes apply via `drizzle-kit push`; auth tables are generated by `@better-auth/cli`, never hand-edited.

The database path is `data/mercury.db`, overridable via `MERCURY_DB_PATH` so tests and CI use scratch databases. The Drizzle handle is cached on `globalThis` because dev hot-reload re-evaluates modules and leaks file handles otherwise.

## Consequences

- Zero-ops persistence; a file copy is a backup.
- Single-writer: fine at this scale, and exam actions additionally guard UPDATEs on `status`/section index against interleaved requests.
- Serverless hosting (Vercel) is off the table without swapping the driver (Turso/libsql or Postgres) — Drizzle keeps that migration mechanical.
- The synchronous driver fits server components/actions well (no connection pool).
